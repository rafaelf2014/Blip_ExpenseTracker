# API Reference

Base URL: `http://localhost:5000/api` (hardcoded in `FrontEnd/src/constants/api.ts`).

All requests/responses are JSON. There is **no authentication** ‚Äî endpoints identify the user by a `userId` in the path or body. CORS is open. The JSON body limit is 5 MB (to allow base64 profile pictures).

Conventions below: `‚Üí` request body, `‚Üê` success response. Error responses are `{ "error": "..." }` with the noted status.

---

## Users & auth

### POST `/api/register`
Create a user.
- ‚Üí `{ username, password }`
- ‚Üê `201 { message }`
- `400` if the username is taken.

### POST `/api/login`
- ‚Üí `{ username, password }`
- ‚Üê `200 { message, user: { id, username } }`
- `401` on invalid credentials.

> The frontend stores `user.id` and `user.username` in `localStorage`; subsequent calls pass `userId`.

### PUT `/api/users/update`
Change a username.
- ‚Üí `{ oldUsername, newUsername }`
- ‚Üê `200 { message }`
- `404` if the user isn't found, `400` if `newUsername` is taken.
- Expenses link by the immutable user `id`, so they need no update.

### PUT `/api/users/update-password`
- ‚Üí `{ username, currentPassword, newPassword }`
- ‚Üê `200 { message }`
- `404` if not found, `401` if `currentPassword` is wrong.

### GET `/api/users/:userId/settings`
Returns the user's financial config, with safe defaults for missing fields.
- ‚Üê `200 { profilePicture, currentBalance, regularTransactions, budgets, balanceHistory, budgetUtilHistory, savingsRateHistory }`
- `404` if the user isn't found.

### PUT `/api/users/:userId/settings`
Partial update ‚Äî only the fields present in the body are written.
- ‚Üí any subset of `{ profilePicture, currentBalance, regularTransactions, budgets, balanceHistory, budgetUtilHistory, savingsRateHistory }`
- ‚Üê `200 { message }`
- `404` if the user isn't found.

## Expenses

### GET `/api/expenses/:userId`
All expense rows for a user (includes generated recurring rows).
- ‚Üê `200 Expense[]`

### POST `/api/expenses`
Add one expense.
- ‚Üí `{ userId, description, amount, category, type, date }`
- ‚Üê `201 { message }`
- `400` if the user isn't found.
- `amount` is coerced with `Number()`; `date` is normalized to ISO. Remember the sign convention: positive = spending, negative = income (see [data-model.md](data-model.md)).

### PUT `/api/expenses/:id`
Update an expense (shallow merge of the body over the existing row).
- ‚Üí any expense fields to change
- ‚Üê `200 { message }`
- `404` if the id isn't found.
- **Side effect:** if the row was generated by a recurring template (`sourceId` present), editing it records a `RecurringSkip` for its original date, so the sync treats that occurrence as user-handled and won't regenerate it.

### DELETE `/api/expenses/:id`
- ‚Üê `200 { message }`
- **Side effect:** same skip behavior as PUT ‚Äî deleting a recurring-generated row records a skip so it isn't recreated.

### POST `/api/expenses/sync-recurring/:userId`
Idempotently materialize recurring templates into real expense rows, up to today.
- ‚Üê `200 { message, created, detached }`
  - `created` ‚Äî number of new occurrence rows added
  - `detached` ‚Äî number of rows orphaned because their template was deleted (their `sourceId` was cleared)
- `400` if the user isn't found.
- Safe to call on every load. See [data-model.md](data-model.md#recurring-transactions) for the full behavior (skip list, detach-on-delete, monthly clamping).

### GET `/api/expense-config`
The category and type vocabularies.
- ‚Üê `200 { categories: string[], expenseTypes: string[] }`
